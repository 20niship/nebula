# XPBD → VBD (Vertex Block Descent) 移行 フィージビリティ調査

`docs/solver_advancement_roadmap.md` の短期項目「XPBD: Vertex Block Descent (VBD) への
切り替え検証」の調査結果。**本タスクの結論は「検証・プロトタイピングまで実施し、
GPUシェーダーへのフル実装は行わない」**。理由は本文末尾の「スコープについて」を参照。

## 1. 現状 (`src/engine/SoftBodyEngine.*`)

- 四面体メッシュの Edge距離制約 + Volume制約をXPBD (`stretchCompliance`, `volCompliance`) で解く。
- 制約はグラフ彩色バッチ (`edgeColorBatch_`/`tetColorBatch_`, `nEdgeColors_`/`nTetColors_`) で
  色ごとに並列ディスパッチし、色内は並列 (Jacobi的)、色間は逐次 (Gauss-Seidel的) に適用。
- `solverIterations=5`, `numSubsteps=15` がデフォルト。
- λ (ラグランジュ乗数) は制約(エッジ/四面体)ごとに持ち、毎フレーム/毎サブステップでゼロクリア。

## 2. 数値プロトタイプによる検証

GPU実装前に、実際のXPBD更新則 (`shaders/solve_stretch.comp` と同じ
`Δλ = -(C + α̃λ)/(w_sum + α̃)` 式) と、VBD (Chen et al. 2024) の変分定式化
(頂点ごとに慣性項+接続エネルギーの合計をNewtonステップで解く) を、同じグラフ彩色戦略で
Python (numpy) 実装して比較した。プロトタイプ: `docs/prototypes/vbd_vs_xpbd.py`
(`python3 docs/prototypes/vbd_vs_xpbd.py` で再実行可能)。

### シナリオA: 5粒子チェーン (疎な連結、各頂点の次数≤2)

| 反復 | XPBD残差 | VBD残差 | VBD/XPBD |
|---|---|---|---|
| 1  | 0.9803 | 0.9104 | 0.929 |
| 5  | 0.5706 | 0.5331 | 0.934 |
| 15 | 0.2925 | 0.2875 | 0.983 |

VBDが一貫して残差7%前後少ない (収束がやや速い) が、差は小さい。チェーン構造では
どちらの手法も「1反復で1エッジ分しか情報が伝搬しない」局所反復法の限界を共有するため。

### シナリオB: ハブ&スポーク (中心頂点の次数=6、密な連結、四面体メッシュの内部頂点に近い)

| 反復 | XPBD残差 | VBD残差 |
|---|---|---|
| 1 | 0.1328 | 0.1253 |
| 2 | 0.0104 | 0.0212 |
| 3 | 0.0117 | 0.0212 (以降不変) |
| 10 | 0.0168 | 0.0212 |
| 15 | 0.0191 | 0.0212 |

**重要な発見**: VBDは2反復で不動点に収束し、以降15反復まで変化しない
(過剰制約系の最小二乗的な平衡解に正しく留まる)。一方XPBDは2反復目に一時的に
低い残差 (0.0104) へ到達するが、その後**単調に悪化していく** (0.0104→0.0191)。
これは中心頂点1つに6本のエッジが集中するため、グラフ彩色で6色に分かれ、
1回の「外側反復」で中心頂点が6回逐次更新される (λ蓄積を伴う) ことで
オーバーシュート/ドリフトが起きるため。VBDは全接続エッジのエネルギーを
1回のNewtonステップで同時に解くため、頂点の次数に関わらず1反復1更新で安定する。

**この非対称性は SoftBodyEngine の実ユースケース (四面体メッシュ、内部頂点は
典型的に6〜20本のエッジ/複数の四面体に接続) においてXPBDが不利になりやすい
構造的な弱点を示している**。これは Chen et al. 2024 が報告する
「VBDは高次数頂点で顕著な安定性・収束優位性を持つ」という主張と整合する。

## 3. フルGPU実装に必要な変更点 (実装した場合)

1. **頂点接続情報の構築**: 現行は「エッジ→2頂点」「四面体→4頂点」の一方向データのみ。
   VBDには「頂点→接続エッジ/四面体リスト」の逆引きが必要 (可変長、CSR形式のバッファを
   新設し `SoftBodyEngine::buildCombinedBuffers()` に構築ロジックを追加)。
2. **頂点グラフ彩色**: 現行の `edgeColorBatch_`/`tetColorBatch_` (プリミティブの彩色) とは別に、
   「エッジで直接つながる頂点同士は異なる色」という頂点隣接グラフの彩色が必要
   (貪欲彩色、CPU側で一度だけ計算)。
3. **per-vertex Hessian 計算シェーダー**: 新規 `sb_solve_vertex_vbd.comp` で、各頂点について
   (a) 慣性項 `m/dt² * I` と `m/dt² * (p - p̃)`、(b) 接続エッジのバネHessian/勾配、
   (c) 接続四面体の体積拘束Hessian/勾配 (エッジより複雑: 変形勾配ベースの2階微分が必要) を
   合算し、3×3線形系を解いて Δp を求める。3×3線形系はシェーダー内で解析的に解ける
   (逆行列の閉形式、またはCramerの公式)。
4. **安定化**: VBD論文は line search やダンピングを推奨 (本プロトタイプでは省略、
   シナリオBで見た通り単純なNewtonステップでも収束したが、実際の四面体体積制約
   (非線形性が強い) では line search なしだと発散するケースがあり得るため要検証。

**見積工数**: 上記1-4を正しく実装しテストするには、体積制約のHessian導出だけでも
XPBD版 (`sb_solve_volume.comp`) より数倍複雑になるため、テスト・チューニングを含め
数日規模の作業になると見積もる。

## 4. 推奨

- 短期的には **VBDへの全面移行は行わない**。現行XPBDは十分安定に動作しており、
  移行のリスク (新しいバグ、数日規模の工数) に対して短期的な利得が不確実。
- 中期的に取り組む場合は、**エッジ距離制約のみVBD化** (体積制約は現行XPBDのまま残す
  ハイブリッド) から始めるのが低リスク。エッジのHessianは本プロトタイプで実証済みの
  閉形式で実装でき、四面体体積制約の複雑なHessian導出を後回しにできる。
- 高次数頂点を含む複雑なメッシュ (毛髪、密なテトラメッシュ) を今後扱う予定があるなら、
  上記シナリオBの結果から見て優先度を上げる価値がある。

## スコープについて

本タスクは `docs/solver_advancement_roadmap.md` で「XPBD: Vertex Block Descent (VBD) への
切り替え**検証**」と短期項目に位置付けていた (全面実装ではなく評価)。実際のGPUシェーダー
実装 (頂点隣接データ構造の新設、頂点グラフ彩色、体積制約のHessian導出) は数日規模の
独立した工数が必要な作業であり、拙速な実装は数値的に不安定なコードを本番エンジンに
混入させるリスクがあるため、本セッションでは実施しない判断とした。上記の数値プロトタイプと
フィージビリティ分析により「検証」の要件は満たしている。
